# Идея 02 — CRM: SLA по стадиям сделки — конкуренты и слабые места

## 1) Матрица конкурентов (6 шт.)

### 1) «Время в стадиях» (Bambit) — `bambit.time_in_stages`
- **Фокус**: время по стадиям (лиды/сделки/счета), скорость обработки (время в первой стадии).
- **Сильные стороны**: подсчёт в рабочих часах; есть выгрузка детализации.
- **Слабые места для SLA**: нет “порогов SLA + эскалаций” как ядра (скорее аналитика).

### 2) «Аналитика скорости CRM» (Aventra) — `aventra.vremya_v_stadiyakh`
- **Фокус**: узкие места, длительность стадий, средний цикл, рейтинг менеджеров, направления сделок, детализация по сущностям.
- **Сильные стороны**: сильное позиционирование “управление скоростью продаж”.
- **Слабые места для SLA**: снова аналитика, а не операционный контроль “просрочено → эскалируй”.

### 3) «История движений по стадиям в сделках и лидах» (Welcm.studio) — `designtut.stagescrm`
- **Фокус**: история переходов по стадиям + длительности (в карточке).
- **Сигналы из отзывов**: встречается критика стабильности (“белый лист”), запросы про смарт‑процессы (зависимость от API).
- **Слабые места для SLA**: нет SLA‑порогов/уведомлений, только “история”.

### 4) «История стадий Сделки» (Fast Bit) — `Sale-Direct.stage_story_deal`
- **Фокус**: история изменений стадий сделки, отчётность по динамике, анализ где “затягивается”.
- **Слабое место**: по отзывам/описанию пользователи часто ждут массовую аналитику “по всем сделкам”, а не просмотр по одной (в подобных решениях это типовой запрос).
- **Слабые места для SLA**: без явных SLA‑правил и алертов.

### 5) «Контроль сроков по сделкам» (Off Group) — `dudoroff.kontrol_srokov_po_sdelkam`
- **Фокус**: таймлайн по сделкам по двум выбранным полям‑датам (старт/финиш), отображает остаток/просрочку.
- **Сигналы из отзывов**: запрос “нет возможности выбрать воронку и отфильтровать по ответственным”.
- **Слабые места для SLA**:
  - это контроль по кастомным полям дат, а не по стадиям/скорости обработки;
  - UX‑запросы: фильтры (воронки/ответственные) критичны.

### 6) «Отчёт PRO Сделки» (Создаватель) — `sozdavatel.bi_deals_boss`
- **Фокус**: большой управленческий дашборд (много вкладок), включает “Скорость”, “Активность”, “Забытые >30 дней”, анализ где застревают.
- **Сильные стороны**: комплексная картина (скорость, качество данных, активность, причины отказов).
- **Слабые места для SLA**:
  - “комбайн” → выше порог внедрения и настройки;
  - SLA‑правила и точечные эскалации могут быть не центральным сценарием.

## 2) Выжимка слабых мест рынка (точки входа для улучшений)
1. **Почти все считают “время/историю”, но не закрывают операционную задачу SLA**: порог → список риска → эскалация → контроль устранения.
2. **Фильтры и разрезы** — главная боль пользователей (воронка/ответственный/период): это нужно делать “из коробки”.
3. **Стабильность и поддержка** — на примере решений “в карточке” важна безупречная работа (иначе 1‑звёздные отзывы).
4. **Слишком широкие права** у многих приложений — можно выиграть доверием (минимальные доступы + прозрачное объяснение).

## 3) Как дифференцироваться в нашей идее (2–3 фишки)
- **SLA‑правила по стадиям** (по воронке/стадии) + понятный “просрочено на X” + эскалации.
- **Пакет действий руководителя**: фильтр “в работе > SLA”, групповой дайджест, “топ причин застревания” (простые метки).
- **Быстрый time‑to‑value**: пресеты SLA (например “первая реакция”, “КП”, “ожидание оплаты”) без сложной настройки.
