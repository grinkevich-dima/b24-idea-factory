# Идея 01 — CRM: SLA реакции на лид

## 1) Коротко (одна фраза)
Панель и алерты, которые показывают **сколько времени проходит до первого касания по лиду** и автоматически эскалируют просрочки.

## 2) Для кого (ICP)
- Руководитель продаж / РОП
- Тимлид колл‑центра / лид‑менеджеров
- Владелец CRM (контроль дисциплины)

## 3) Боль и JTBD
- Лиды “висят” без реакции → падает конверсия и доверие к каналу.
- РОП не видит, кто тормозит: менеджер/очередь/перераспределение.

## 4) Измеримый эффект
- Снижение доли лидов без первого касания > N минут/часов.
- Рост конверсии лид → сделка (как вторичная метрика).

## 5) УТП (позиционирование)
**“Покажем и сократим время до первого касания по лидам за 10 минут после установки.”**

## 6) MVP (3–5 фич)
1. **Дашборд SLA**: лиды в разрезе источников/ответственных/статусов SLA (ОК/в зоне риска/просрочено).
2. **Правила SLA**: пороги по источнику/времени суток/рабочим часам (минимально).
3. **Эскалации**: уведомление менеджеру → руководителю при просрочке.
4. **Топ нарушителей**: рейтинг по просрочкам и среднему времени реакции.
5. (Опционально) **Экспорт** отчёта (CSV/XLSX) или “поделиться ссылкой” (если уместно в B24).

## 7) Реализуемость и тех‑чеклист (10 минут)
### Данные и объекты
- Сущности: **лид** (обяз.), возможно **делка** (для конверсии).
- Нужные поля: дата создания, ответственный, источник, статус/стадия.

### “Первое касание” — как определить (сделать настройкой)
В MVP выбрать 1–2 простых и понятных правила, например:
- первое **исходящее** дело/активность по лиду; или
- первая смена стадии “В работе”; или
- первая исходящая коммуникация (если доступно без широких прав).

### Триггеры и обновление
- Достаточно **инкрементального пересчёта** по событиям изменения лида/дел (без постоянного мониторинга).
- Допустимая задержка: минуты.

### Права и безопасность
- Минимизировать доступ: читаем лиды в пределах прав пользователя/роли.
- Не хранить тексты коммуникаций (только метрики/времена).

### Производительность
- Считать только лиды “за период” (например, последние 7/30 дней) + фильтры.
- Для больших порталов — кэш агрегатов по дням/источникам.

### UI‑объём
- Экран 1: дашборд.
- Экран 2: правила SLA (простая форма).
- Экран 3 (опц.): детализация по конкретному менеджеру/источнику.

## 8) Риски (и как снизить)
- **Неочевидность определения “касания”** → сделать настройкой + дефолт “перевод в работу”.
- **Широкие права ради коммуникаций** → в MVP считать по изменению стадий/активностей без чтения содержимого.
- **Шум от алертов** → “тихие окна”, гистерезис, группировка уведомлений.

## 9) Метрики успеха
- Time‑to‑value: пользователь видит просрочки и топ‑лиды за ≤10 минут.
- Доля лидов с реакцией ≤ SLA.
- Среднее время реакции (по источникам/менеджерам).

## 10) Поиск конкурентов (ключевые запросы для Маркета)
- “SLA лид”
- “скорость обработки лидов”
- “время реакции на лид”
- “контроль лидов”
- “KPI лид менеджера”
- “просроченные лиды”
- “первый контакт лид”

## 11) Упаковка для Маркета (черновик)
### Название
SLA по лидам: контроль скорости реакции

### Буллеты ценности
- Видите лиды, которые “простаивают”, до потери клиента
- Настраиваемые пороги SLA по источникам
- Авто‑эскалации просрочек руководителю
- KPI по менеджерам и источникам в одном экране
