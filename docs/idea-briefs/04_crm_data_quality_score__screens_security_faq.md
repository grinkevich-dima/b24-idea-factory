# Идея 04 — CRM: Data Quality Score — скрины, безопасность, FAQ

## 1) План скриншотов (6–8)
1. **Дашборд Data Quality**\n
   - общий скоринг (%), критичные пропуски, динамика.\n
   - фильтры: период, стадия, ответственный.
2. **Список нарушений**\n
   - сущность, стадия, ответственный, какие поля не заполнены.\n
   - переход в карточку.
3. **Конструктор профиля требований**\n
   - выбор сущности (старт: сделки или лиды)\n
   - выбор стадии → список обязательных полей.
4. **Пресеты “из коробки”**\n
   - 2–3 готовых профиля (например: “минимум”, “строго на финале”).\n
5. **Напоминания**\n
   - частота, лимиты, кому эскалировать.\n
6. (Опц.) **Предупреждение при переводе стадии**\n
   - пример уведомления “не заполнены поля…”.

## 2) Чек‑лист модерации и безопасности
### Права
- **CRM**: чтение выбранной сущности и метаданных полей.\n
- **Настройки пользовательских полей**: запрашивать только если реально нужно читать/создавать поля.\n
  В MVP лучше **не создавать поля автоматически**, а работать с уже существующими.
- **Чат и уведомления**: отправка уведомлений (без чтения сообщений).\n
- **Пользователи (минимальный)**: отображение сотрудников.

### Хранение
- Не хранить значения полей.\n
- Хранить только “поле заполнено/нет” + идентификатор сущности/стадии + агрегаты.

### UX‑политика
- В MVP **никаких жёстких блокировок**; только подсказки/отчёты/напоминания.

## 3) FAQ (готовый текст)
**Храните ли вы значения полей?**\n
Нет. Мы фиксируем только факт заполненности (да/нет) и строим отчёты.

**Можно ли начать с одного процесса?**\n
Да, вы можете включить контроль только для одной воронки/стадий и постепенно расширять.

**Будет ли приложение мешать менеджерам?**\n
Нет: по умолчанию это “мягкий контроль” (отчёт и напоминания), без блокировок.
