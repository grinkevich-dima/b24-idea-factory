# Идея 03 — CRM: “Сделки без касаний” — конкуренты и слабые места

## 1) Матрица конкурентов (4–5 шт.)

### 1) «Отчёт PRO Сделки» (Создаватель) — `sozdavatel.bi_deals_boss`
- **Фокус**: большой управленческий дашборд; в “Активности” есть сегменты:\n
  - “Забытые >30 дней — сделки без активности более 30 дней”\n
  - активные/горячие/закрытые.
- **Как считают активность**: “любое изменение в сделках — дела, письма, звонки, заполнения полей и т.д.”
- **Слабые места для нашей идеи**:
  - это часть большого отчёта (порог входа/настройки/много вкладок);
  - “без касаний” не выделено как отдельный быстрый сценарий “вернуть в работу”.

### 2) «Контроль забытых клиентов» (SalesX) — `salesx.app7`
- **Фокус**: “верните в работу сделки, о которых забыли — за 10 секунд”; список “забытых” по периоду 1/3/7/14/30 дней.
- **Сильные стороны**:
  - фильтры по менеджеру, воронке, стадии, сумме, дате активности;
  - приоритизация и режим “Фокус” (TOP‑10);
  - быстрые действия (позвонить/написать/создать задачу/напоминание).
- **Слабые места/риски**:
  - новый продукт (мало установок/нет отзывов) → недостаточно доверия рынку;
  - запрос полного доступа к CRM (как и у большинства подобных решений).

### 3) «Дата последней активности клиента (гибкий фильтр)» (СкайВеб24) — `skyweb24.lastactivity`
- **Фокус**: записывает “дату последней коммуникации” в пользовательское поле и позволяет фильтровать/сортировать стандартными списками.
- **Сильные стороны**:
  - хорошо для больших баз: есть “постепенная обработка”, режим BI‑конструктора или REST‑режим;
  - позиционирование “без передачи данных на сторонние серверы”.
- **Слабые места для нашей идеи**:
  - это “строительный блок” (поле/фильтр), но не готовый процесс “список забытых + действия + эскалации”.

### 4) «Дата последней сделки у компании и контакта» (Welcm.studio) — `designtut.dealdate`
- **Фокус**: поле “дата последней успешной сделки” у компаний/контактов, пригодно для фильтров/роботов/триггеров.
- **Сильные стороны**: очень простой сценарий, заявлен “без отправки данных на сторонние серверы”.
- **Слабые места для нашей идеи**:
  - считает по успешным сделкам, не по активности в текущих сделках;
  - закрывает ретеншн/реактивацию, а не “не дожимаем текущие сделки”.

## 2) Слабые места рынка (куда целиться)
1. **Либо “комбайн” отчётов, либо узкое поле‑утилита** — не хватает “узкого, но завершённого” продукта: список → приоритизация → действия → контроль результата.
2. **Определение “активности” размыто**: пользователям нужно простое и настраиваемое правило (что считать касанием).
3. **UX‑победа**: TOP‑10 + быстрые действия + дайджест руководителю часто важнее “ещё одного отчёта”.
4. **Доверие**: много новых/малых приложений без отзывов — можно выиграть прозрачностью, стабильностью, быстрыми релизами.

## 3) Как дифференцироваться в нашей идее (2–3 фишки)
- **Двойная модель “касания”**: “изменение сделки” vs “коммуникации/дела” (переключатель), плюс исключения по стадиям.
- **Контур результата**: пометка “возвращено в работу” (после активности) и отчёт “сколько реанимировали”.
- **Режим руководителя**: ежедневный дайджест “топ забытых” по каждому менеджеру + эскалация при игноре.
