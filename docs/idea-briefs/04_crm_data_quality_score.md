# Идея 04 — CRM: Контроль заполненности карточек (Data Quality Score)

## 1) Коротко (одна фраза)
Скоринг качества данных в CRM: показывает **какие обязательные поля не заполнены** (по стадиям/ролям) и помогает дисциплинировать команду.

## 2) Для кого (ICP)
- Владелец CRM / администратор
- РОП (качество воронки и отчётов)
- Руководитель контакт‑центра/лидогенерации

## 3) Боль и JTBD
- Поля заполняют “как получится”, отчёты и аналитика становятся недостоверными.
- При масштабировании CRM превращается в “свалку данных”.

## 4) Измеримый эффект
- Рост заполненности ключевых полей (по сделкам/лидам).
- Снижение доли карточек с критичными пропусками на поздних стадиях.

## 5) УТП (позиционирование)
**“Порядок в CRM за неделю: прозрачные требования к полям + контроль нарушений + мягкие напоминания.”**

## 6) MVP (3–5 фич)
1. **Профили требований**: набор обязательных полей по сущности (лид/сделка/контакт/компания) и стадии.
2. **Скоринг**: % заполненности + список критичных пропусков.
3. **Отчёт нарушений**: по ответственным/стадиям/профилям.
4. **Напоминания**: уведомления ответственным, эскалация руководителю при систематике.
5. (Опционально) **“Мягкая блокировка”**: предупреждение при попытке перевести на стадию при неполном наборе (если это допустимо UX-ом).

## 7) Реализуемость и тех‑чеклист (10 минут)
### Данные и объекты
- Сущности: лид/сделка (начать с одной, лучше со **сделки** или **лида**).
- Поля: стандартные + пользовательские (должны быть выбираемыми из списка полей).

### Триггеры и обновление
- Пересчёт при изменении сущности/стадии.
- Для отчётов — периодический пересчёт агрегатов (раз в сутки/час).

### Права и безопасность
- Настройка профилей — роль “админ/владелец CRM”.
- Просмотр нарушений — в рамках прав на сущности.
- Не хранить значения полей, хранить только “заполнено/нет” и идентификаторы.

### Производительность
- Не делать полный скан “всего за всё время”: период/активные/в работе.
- Агрегаты по ответственным и стадиям.

### UI‑объём
- Экран 1: дашборд качества/нарушений.
- Экран 2: конструктор профиля требований (простая форма).

## 8) Риски (и как снизить)
- **Слишком много вариативности полей** → начать с 1 сущности и 1–2 профилей “из коробки”.
- **Негатив от “блокировок”** → в MVP только предупреждения и отчёт (без жёсткого запрета).
- **Конфликты прав** → показывать нарушения только по доступным сущностям.

## 9) Метрики успеха
- % карточек без критичных пропусков на ключевых стадиях.
- Кол-во нарушений на пользователя (до/после).
- Время “до исправления” после напоминания.

## 10) Поиск конкурентов (ключевые запросы для Маркета)
- “контроль заполнения полей”
- “качество данных CRM”
- “обязательные поля по стадиям”
- “валидация полей сделки”
- “data quality”
- “CRM дисциплина”

## 11) Упаковка для Маркета (черновик)
### Название
Data Quality для CRM: контроль обязательных полей

### Буллеты ценности
- Прозрачные правила заполнения по стадиям
- Скоринг качества данных и список нарушений
- Напоминания и отчёты по менеджерам
- Не ломает процесс: можно начать с мягкого контроля
