# Идея 05 — Задачи: “Риск проекта за 1 экран”

## 1) Коротко (одна фраза)
Дашборд, который агрегирует сигналы (просрочки, блокеры, перегруз) и показывает **риск срыва проекта/направления задач**.

## 2) Для кого (ICP)
- Руководитель проектов / PM
- Руководитель отдела / PMO
- Тимлид (контроль выполнения)

## 3) Боль и JTBD
- “Где горит?” видно слишком поздно, пока проблема уже не стала кризисом.
- Статусы субъективны, а метрики разрознены.

## 4) Измеримый эффект
- Раннее выявление зон риска (до массовых просрочек).
- Снижение просрочек и “зависших” блокеров.

## 5) УТП (позиционирование)
**“Один экран вместо 10 отчётов: риски проекта по задачам — прозрачно и заранее.”**

## 6) MVP (3–5 фич)
1. **Сводка риска** по проекту/группе задач: низкий/средний/высокий (простые правила).
2. **Топ сигналов**: просроченные, без дедлайна, блокирующие, задачи без ответственного, перегруз исполнителя.
3. **Пороги и настройки**: что считать риском (N просрочек, % просроченных, WIP‑порог на человека).
4. **Уведомления**: если риск вырос до “высокий” → алерт PM/руководителю.
5. (Опционально) **Тренд** риска по дням/неделям.

## 7) Реализуемость и тех‑чеклист (10 минут)
### Данные и объекты
- Сущность: **задача** (проект/группа как разрез).
- Поля: дедлайн, статус, ответственный, проект/группа, связи (если есть), дата изменения.

### Триггеры и обновление
- Можно считать “по запросу” + периодический пересчёт (например, раз в 1–3 часа).
- Инкрементально обновлять агрегаты при изменении статуса/дедлайна.

### Права и безопасность
- Отчёт в рамках прав пользователя на задачи/проекты.
- Не нужно хранить содержимое, достаточно агрегатов.

### Производительность
- Не сканировать “всё”: только активные задачи/период/конкретный проект.
- Кэш агрегатов по проекту.

### UI‑объём
- Экран 1: дашборд риска + сигналы.
- Экран 2: настройки порогов.

## 8) Риски (и как снизить)
- **Слишком умная модель риска** → в MVP только правило‑based (прозрачно и объяснимо).
- **Разные процессы у компаний** → дать пресеты “по умолчанию” + настройку порогов.
- **Шум от алертов** → изменения уровня риска, а не каждый инцидент.

## 9) Метрики успеха
- Кол-во проектов с высоким риском (динамика).
- % просроченных задач по проектам.
- Среднее время “до реакции” после алерта.

## 10) Поиск конкурентов (ключевые запросы для Маркета)
- “дашборд задач”
- “контроль просрочек”
- “риск проекта”
- “KPI задач”
- “перегруз исполнителей”
- “WIP лимиты”
- “блокеры задач”

## 11) Упаковка для Маркета (черновик)
### Название
Риск проекта по задачам: дашборд и сигналы

### Буллеты ценности
- Риск проекта в одном экране (просрочки, блокеры, перегруз)
- Настраиваемые пороги и пресеты “из коробки”
- Алерты при росте риска
- Подходит руководителю и PM без сложной настройки
