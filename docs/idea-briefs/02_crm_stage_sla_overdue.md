# Идея 02 — CRM: SLA по стадиям сделки (застревания)

## 1) Коротко (одна фраза)
Дашборд и алерты, которые показывают **где сделки застревают по стадиям**, и автоматически эскалируют превышение SLA.

## 2) Для кого (ICP)
- РОП / руководитель отдела продаж
- Руководитель группы (тимлид)
- Владелец CRM (контроль процесса)

## 3) Боль и JTBD
- Сделки зависают на стадиях, причина неочевидна.
- Управление “по ощущениям”: нет единого экрана просрочек по этапам.

## 4) Измеримый эффект
- Снижение количества сделок с превышением SLA по стадиям.
- Уменьшение медианного времени прохождения воронки.

## 5) УТП (позиционирование)
**“Одна панель показывает просрочки по стадиям и помогает ускорить воронку без ручного контроля.”**

## 6) MVP (3–5 фич)
1. **Настройка SLA**: пороги по стадиям (дни/часы), опционально по типу/направлению сделки.
2. **Дашборд просрочек**: список просроченных сделок + сводка по стадиям (сколько и насколько превышено).
3. **Эскалации**: уведомления ответственному и руководителю при выходе за порог.
4. **Причины/метки** (лёгкая версия): пользователь может пометить “почему застряло” из фиксированного списка (без сложной аналитики).
5. (Опционально) **Экспорт** отчёта.

## 7) Реализуемость и тех‑чеклист (10 минут)
### Данные и объекты
- Сущность: **сделка**.
- Нужные поля: стадия, дата изменения стадии (или аналог), ответственный, направление/воронка.

### Триггеры и обновление
- Идеально: событие **смены стадии** → обновить “время входа в стадию”.
- Для надёжности: периодическая проверка “просрочено ли сейчас” по активным сделкам (например раз в 15–60 минут) без полного скана (по фильтрам: активные/в процессе).

### Права и безопасность
- Доступ только к сделкам, которые пользователь вправе видеть.
- Не храним чувствительные данные, достаточно метрик/времени.

### Производительность
- Считать только “активные” сделки и период (например, последние 90 дней).
- Сводки строить агрегатами (по стадиям/ответственным), кэшировать.

### UI‑объём
- Экран 1: дашборд/список.
- Экран 2: настройка SLA.

## 8) Риски (и как снизить)
- **Разные воронки/стадии у клиентов** → SLA настраивается по выбранной воронке, дефолты “выключено”.
- **Шумные уведомления** → объединение в дайджест, “тихие часы”, лимит алертов.
- **“Причины застревания” превращаются в большой проект** → в MVP только короткий список меток и отчёт по ним.

## 9) Метрики успеха
- Доля активных сделок, превысивших SLA.
- Время на стадии (P50/P90) по ключевым стадиям.
- Количество устранённых просрочек после алертов.

## 10) Поиск конкурентов (ключевые запросы для Маркета)
- “SLA по сделкам”
- “контроль стадий сделки”
- “застревание сделок”
- “просроченные сделки”
- “дашборд продаж”
- “KPI по стадиям”
- “воронка скорость прохождения”

## 11) Упаковка для Маркета (черновик)
### Название
SLA по стадиям сделок: просрочки и ускорение воронки

### Буллеты ценности
- Видите просроченные стадии в одном экране
- Настраиваемый SLA по этапам и воронкам
- Эскалации ответственным и руководителям
- Минимальная настройка, быстрый time‑to‑value
