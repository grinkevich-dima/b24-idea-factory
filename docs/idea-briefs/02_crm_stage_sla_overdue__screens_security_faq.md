# Идея 02 — CRM: SLA по стадиям сделки — скрины, безопасность, FAQ

## 1) План скриншотов (6–8)
1. **Сводка по стадиям (heatmap/таблица)**\n
   - стадия → кол-во просрочек → среднее превышение → максимум.\n
   - фильтр по воронке/периоду.
2. **Список просроченных сделок**\n
   - сделка, стадия, ответственный, сколько дней на стадии, превышение SLA.\n
   - переход в карточку сделки.
3. **Настройка SLA по стадиям**\n
   - экран “воронка → список стадий → пороги”.\n
   - пресет “быстро настроить” (3–4 шаблонных порога).
4. **Эскалации/дайджест**\n
   - кому отправлять, как часто, режим “только если просрочек > N”.
5. **Детализация стадии**\n
   - “в этой стадии просрочено” → топ сделок + топ ответственных.
6. **Метки причин (если включены)**\n
   - список причин + отчёт “частота причин по стадиям”.

## 2) Чек‑лист модерации и безопасности
### Права (минимально)
- **CRM**: чтение сделок, стадий/воронок, ответственных.\n
- **Пользователи (минимальный)**: отображение сотрудников.\n
- **Чат и уведомления**: отправка уведомлений (без чтения сообщений).

### Что не хранить
- Содержимое карточек/комментариев.\n
- Любые “письма/сообщения” — только метрики времени.

### Хранение
- Времена входа в стадию/последней смены стадии, SLA‑порог, статус просрочки.\n
- Ограничить период (например 90–180 дней).

## 3) FAQ (готовый текст)
**Что считает приложение?**\n
Время нахождения сделок в каждой стадии и факт превышения заданного SLA.

**Нужно ли долго настраивать?**\n
Нет: выберите воронку и задайте пороги по стадиям. Можно начать с пресета и уточнить позже.

**Как работают уведомления?**\n
Уведомления отправляются при превышении SLA, можно включить дайджест и лимиты, чтобы не было “шума”.
