# Bitrix24 Idea Factory (конвейер ≤5 часов на 1 приложение)

## Ограничения (обязательные)
- Приложения: **публичные** для **Bitrix24 Marketplace**
- Домены: **CRM**, **Задачи/Проекты**, **Аналитика/Отчёты**, **Чаты**
- Исключить: **внешние интеграции**, **отраслевые решения**
- Язык артефактов: **русский**

## Выходные артефакты (что создавать в репозитории)
Папка: `docs/idea-briefs/`

### Индекс
- Всегда создавать/обновлять: `docs/idea-briefs/INDEX.md`

### Пакет по одной идее
Для каждой выбранной идеи `NN` (01, 02, …) создавать 4 файла:
- `docs/idea-briefs/NN_<slug>.md`
- `docs/idea-briefs/NN_<slug>__competitors.md`
- `docs/idea-briefs/NN_<slug>__mvp_release_market.md`
- `docs/idea-briefs/NN_<slug>__screens_security_faq.md`

`<slug>` должен быть кратким и стабильным (латиница + подчёркивания), описывать суть.

## Конвейер (строго по шагам)

### Шаг 0 — Подготовка (10 минут)
Сформировать 1 предложение:\n
**“Для кого + где в Bitrix24 боль + какой измеримый эффект”**.

### Шаг 1 — Генерация 15–25 идей (35 минут)
Для каждой идеи зафиксировать минимум:
- ICP/роль покупателя
- боль (1–2 предложения)
- измеримый эффект (метрика)
- MVP‑объём (на 1–2 недели)
- основные риски

Сразу отсеивать идеи, которые требуют внешних интеграций или отраслевой специфики.

### Шаг 2 — Скоринг, реализуемость и техсложность (20 минут)
Оценка 1–5 (3–5 минут на идею), выбрать **Top‑5**.

**A — рынок/ценность**\n
- частота боли\n
- платёжность/ROI\n
- дифференциация

**B — реализуемость (feasibility)**\n
- опора на стандартные объекты (CRM/Tasks/IM)\n
- понятный источник данных\n
- минимальные права\n
- без сложной инфраструктуры

**C — техсложность (MVP ≤1–2 недели)**\n
- UI: 1 killer‑экран + 1–2 настройки\n
- минимум тяжёлых фоновых процессов\n
- производительность на больших порталах\n
- управляемые edge‑cases

**Красные флаги (дисквалификация)**\n
- постоянный realtime/стриминг/сложные боты в чате\n
- “админ‑всё” без ясной ценности\n
- тяжёлая инфраструктура/очереди/внешние хранилища\n
- нельзя обеспечить time‑to‑value ≤ 10 минут

### Шаг 3 — Анализ конкурентов (70 минут)
Для каждой из Top‑5:
- найти 5–10 конкурентов в Marketplace по 10–15 ключевым запросам (синонимы и смежные формулировки)
- собрать по каждому: обещание, механика, installs/оценка (если видны), отзывы (сигналы болей), запрашиваемые права, пробелы
- итог: топ слабых мест рынка + “что мы делаем лучше”

Записать результат в `__competitors.md`.

### Шаг 4 — УТП + MVP + релиз + текст Маркета (75 минут)
Сформировать:
- **УТП** (1 предложение)
- **MVP 3–5 фич** (строго)
- **Не‑MVP** (что выкидываем, чтобы не распухло)
- план релиза **Неделя 1 / Неделя 2**
- метрики успеха + риски/смягчения
- черновик текста для Маркета: заголовок, подзаголовок, буллеты ценности, quick start, короткий FAQ

Записать в `__mvp_release_market.md`.

### Шаг 5 — Скриншоты + безопасность/модерация + FAQ (70 минут)
Сформировать:
- план 6–8 скриншотов (что на каждом показываем)
- стратегия минимальных прав
- что приложение **не читает/не хранит** (особенно контент сообщений)
- политика хранения (только агрегаты/timestamps)
- производительность (периоды по умолчанию, кэширование)
- FAQ (готовый текст “вставить в Маркет”)

Записать в `__screens_security_faq.md`.

## Правила качества (обязательные)
- Не делать “комбайны” из 8 вкладок в MVP: один главный экран + короткие настройки.
- Побеждать доверием: минимальные права + прозрачное описание безопасности.
- Всегда обеспечивать быстрый старт: **польза ≤ 10 минут после установки**.
